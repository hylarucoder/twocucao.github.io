发表于 2017-12-07 | 分类于 学习资源
| 字数统计 1,725 | 阅读时长 7

0x00 前言

机缘巧合，最近公司突然要搞一波大量数据的分析。属于进店客流类的分析，原始数据非常大。 数据量级也还算不错，经过 gzip 压缩，接近 400 个 点位的 SQL 文件 (MySQL innoDB)，大小接近 100GB 左右，原始记录数据估测在180亿左右。

解压后…… 差不多一个 T 吧。

如果是人民币玩家，自然是直接购置几十台高配置机器，做个 mysql shard 或者直接上大数据全家桶比如 hadoop 和 hive 之类，让程序员去往死里折腾吧。

嗯，然而对于我这种非人民币玩家，就要用单机硬扛。

我的配置简单来说如下：

  • 局域网服务器 （ Ubuntu 16.04 LTS ）
    • Xeon(R) CPU E3-1225 v5 @ 3.30GHz
    • 16G 内存
    • 1T 硬盘
  • 苹果电脑 2016 年 15寸 最高配
    • 1T 硬盘
    • i7 四核

0x01 准备数据阶段

用低配机器分析大数据的首要原则，就是不要分析大数据。

换而言之就是尽可能的抽取所得结论所需分析数据的最小超集, 小机器是海量计算的，但通过一定的过滤和筛选可以将数据筛选出到一台机器能扛得住的计算量。

1.1 将数据导入 MySQL 中

我们先不管三七二十一，既然给了 SQL 文件，肯定要入库的，那么问题来了：

将大象关进冰箱要几个步骤 将数据导入数据库中需要几个步骤

或者说，如何更快的导入300余张不同表的数据。

大致步骤如下：

  • 初始化硬盘
  • 配置 MySQL
  • 解压文件 (7za x)
  • 导入数据 (PV & MySQL)
  • 删除已完成数据包，节省磁盘空间 (rm)

第二步和第四步无需解释。提一下配置MySQL和导入数据包两个步骤。

导入数据 之 MySQL + PV

我们使用 mysql 导入脚本的时候，有几种导入方式

  • source 命令，然而这个命令容易在数据量很大的时候直接卡掉。（印象中是直接把 sql 文件加载到内存中，然后执行）
  • mysql 命令

加上 PV 命令的话，比较神奇了。有进度条了！!

然后，可以查看一下磁盘 CPU 内存的占用情况，如果可能的话，使用 tmux 多开几个进程导入数据。

因为每个 SQL 文件对应的表不一样，所以多开几个进程批量insert的话并不会锁表，这样可以显著提升导入速度。

1.2 导出数据

过滤掉不需要的行与列

尽量拆分数据为最小单元的 CSV 文件

经过初步观察，数据量特别大。如果把所有的数据拿出来进行计算的话，距离算出结果就遥遥无期了。

其实这个时候给 CSV 文件进行分析会比给 SQL 分析会好很多。

PS : 如果每个文件的数据量不是很大的话，直接给 CSV 文件进行分析会快很多很多。

0x02 分析阶段

经过上一步处理，数据的文件总大小大约从 1000GB -> 10GB 左右，然后拆分成若干个文件。

  • 分析性能要点1：简单的统计用读写文件系统是最快的
  • 分析性能要点2：拆分数据成若干个文件,使得自己可以在比较短的时间内读取。
  • 分析性能要点3：排序后的list和set谁查找速度最快？
  • 分析性能要点4：reduce 加 python 里面唯一的set update的时候是不是用空间换时间。可用空间换时间也会带来内存不够的情况下，空间换时间效果也不好。
